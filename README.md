## 核心技术要点（面试重点）
### 1. 为什么用`RecordAdd`继承`BaseModel`，而不是直接用字典接收参数？
- **类型安全校验**：通过`amount: float`明确字段类型，自动拦截字符串、布尔值等错误类型的入参，避免非法数据进入业务逻辑。
- **参数安全校验**：配合`Field`的`ne=0`、`min_length=1`等规则，在请求到达接口前自动完成校验，无需手写大量if判断逻辑。
- **FastAPI原生集成**：自动在`/docs`接口文档中生成完整的入参结构、说明和示例，前端对接无需额外沟通，降低联调成本。
- **代码可维护性**：参数结构集中管理，后续新增/修改字段只需修改这一个类，无需逐个修改接口的参数处理逻辑。
- **标准化数据解析**：天然支持JSON格式解析，前端传入的JSON数据可直接映射为Python对象，无需手动做字典转对象的处理。

### 2. `amount`字段为什么要加`ne=0`的校验？
- **业务目的**：记账场景中金额为0无实际业务意义（正数为收入、负数为支出），从接口入口处直接拦截无效数据，避免脏数据入库。
- **不加的潜在问题**：会产生无意义的冗余数据，干扰月度收支统计、收支趋势分析等核心功能的准确性，同时降低接口的容错性。
- **代码实现**：
```python
amount: float = Field(
    ...,
    ne=0,  # 核心校验：金额不能为0
    description='记账金额(单位元，正数为收入，负数为支出)',
    example=100.5
)
```

### 3. 为什么把remark字段设为Optional[str]并设置默认值'无'，和直接写remark: str = '无'有什么区别？
- **明确为非必填**:Optional[str]从类型层面明确标注该字段为非必填项，前端请求时可以不传这个字段，接口不会报错；同时设置默认值'无'，不传时自动填充默认值，兼顾灵活性和数据完整性。
- **两种写法的核心区别**：Optional[str]符合 Python 类型提示规范，会在接口文档中明确标注为非必填，不会给前端和维护者造成误解；仅写remark: str = '无'，虽然有默认值，但类型层面仍标注为必填字符串，会误导对接方，也不符合 Pydantic 的最佳实践。
- **代码实现**：
```python
remark: Optional[str] = Field(
    '无',
    description="记账备注（可选）",
    example="午餐吃了牛肉面"
)
```

---

### 4. 为什么给每个字段的Field里都加了description和example参数？
- **自动生成标准化接口文档**：description和example会自动同步到 FastAPI 的/docs在线文档中，前端开发者可以直接看到每个字段的含义和正确的填写示例，无需额外的接口文档沟通。
- **统一入参规范**：通过example给出标准的入参格式，能统一前后端的入参标准，减少联调时的格式分歧。

---

### 5.为什么给category字段设置min_length=1的校验？
- **拦截空字符串**：从接口入口处拦截空字符串、纯空格的无效分类参数，确保每一条记账数据都有合法的业务分类，保证入库数据的有效性。
- **不加校验的潜在问题**：无效空值会直接入库，干扰统计结果；同时需要在业务代码中额外手写非空判断逻辑，增加代码冗余。
- **代码实现**：
```python
category: str = Field(
    ...,
    min_length=1,
    description='消费/收入分类（如餐饮、工资）',
    example='餐饮'
)
```
